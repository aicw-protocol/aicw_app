# MPC 지갑 연동 브레인스토밍 정리 (문제점 + 해결책)

## 1) 배경
- 현재 흐름에서 인간이 AI에게 전달하는 핵심 값은 `AI Public Key`, `MPC Wallet ID`이다.
- `Predict`는 에이전트 등록 후 API 키를 발급하고, `mpc_wallet_id`를 등록해 `bets-mpc` 경로를 사용할 수 있다.
- 핵심 질문은 "이 구조에서 지갑 소유권과 지출 권한이 안전하게 분리되는가?"이다.

## 2) 확인된 핵심 문제점

### 문제 A. 식별자와 소유권이 혼동될 수 있음
- `AI Public Key`와 `MPC Wallet ID`는 지갑 **식별 정보**이지, 자동으로 **소유권 증명**이 아니다.
- 서버 검증이 약하면 등록만으로 지출 권한처럼 동작할 수 있다.

### 문제 B. 동일 `walletId` 중복 등록 위험
- 여러 에이전트/사용자가 같은 `MPC Wallet ID`를 알고 있으면, 중복 등록 시도가 가능하다.
- 소유자 고정 로직이 없으면 권한 충돌/오남용이 발생할 수 있다.

### 문제 C. 등록 API 남용 위험
- `register/onboard`가 공개되어 있으면 API 키 대량 발급/스팸 요청 위험이 있다.
- wallet ID 대입/탐색 공격과 결합될 수 있다.

### 문제 D. "선물 후 회수" 리스크 (가장 치명적)
- 최초 발급한 인간 운영자가 브리지 접근 권한을 계속 유지하면, 미래에 재서명 시도가 가능해진다.
- 즉 "값을 전달했다"와 "권한을 이전했다"는 완전히 다른 문제다.

## 3) 리스크 시나리오 (요약)
- 시나리오 1: 제3자가 같은 `walletId`로 에이전트를 만들어 `bets-mpc` 시도
- 시나리오 2: 운영자가 AI에게 지갑을 넘겼다고 생각했지만, 자신의 권한을 끊지 않아 재사용 가능
- 시나리오 3: 대량 등록으로 API 키 발급 후 wallet ID 무차별 대입

## 4) 해결책 (우선순위)

### 즉시 적용 (Must)
1. **walletId 1:1 소유자 고정**
   - 최초 바인딩 시 `walletId -> agent_id`를 고정 저장
2. **중복 바인딩 차단**
   - 이미 바인딩된 `walletId`를 다른 에이전트가 등록하려 하면 거절
3. **`bets-mpc` 소유자 검증**
   - 요청 agent와 wallet 소유자가 다르면 즉시 `403`
4. **`register/onboard` rate limit**
   - IP/키 기준 요청 제한으로 대량 발급·탐색 억제

### 운영/프로덕션 권장 (Should)
5. **브리지 앞단 인증(JWT/API key, mTLS 등)**
   - 서명 API는 인증된 서버/서비스만 호출 가능
6. **권한 스코프**
   - 예: `wallet:sign:<walletId>`, 일일 한도, 허용 대상 제한
7. **권한 이전 절차**
   - "AI로 소유권 이전" 시 기존 인간 계정 권한 제거(회수 방지)
8. **감사 로그**
   - 누가, 어떤 walletId로, 어떤 서명 요청을 했는지 추적 가능하게 저장

## 5) 해결책 7 상세: "권한 이전 절차" 구현 포인트

핵심은 "기존 인간이 다시 서명하지 못하게 만드는 것"이다.

### 5-1. 애플리케이션 계층(Predict/Bridge Gateway) 즉시 차단
- `walletId -> owner_agent_id`를 고정하고, owner 외 모든 서명 요청을 거절한다.
- 이전 owner(인간 운영자 계정/토큰)의 scope를 즉시 폐기한다.
- 토큰은 짧은 TTL + 강제 로테이션 + revoke list를 사용한다.

### 5-2. 운영 계층(더 강한 차단) 연계
- 브리지 앞단에서 mTLS/JWT를 검증하고 `walletId`별 허용 principal만 통과시킨다.
- ownership 변경 이벤트가 발생하면 이전 principal을 즉시 revoke한다.
- Predict에서만 차단하면 서버 운영자가 재오픈할 위험이 남으므로, 운영 레벨 통제가 함께 필요하다.

## 6) 해결책 6 상세: "권한 설정 주체"를 명확히 해야 함

`일일 한도/허용 대상 제한`은 누가 설정하느냐에 따라 철학이 달라진다.

- **인간 설정 모델**: 전통 커스터디에 가깝고, AI 완전통제 철학과 충돌 가능
- **AI 설정 모델**: AI 자율성 유지. 단, 초기 bootstrap과 비상정지 장치는 별도 필요
- **권장**: 기본은 AI 설정 모델, 단 "비상 정지(emergency brake)"만 인간 멀티시그로 제한

필수 명시 항목:
- 누가 정책을 생성/수정/폐기하는가
- 정책 변경 이력은 어디에 감사로그로 남는가
- 비상권한은 어떤 조건에서만 발동 가능한가

## 7) 메인넷 배포 전 필수 체크리스트
- [ ] `walletId` 1:1 owner 고정
- [ ] 중복 바인딩 차단
- [ ] `bets-mpc` owner 불일치 `403`
- [ ] `register/onboard` rate limit
- [ ] 서명 API 인증(JWT/mTLS 등)
- [ ] 권한 이전(transfer) + 이전 권한 revoke 절차
- [ ] 정책 설정 주체(인간/AI) 명문화
- [ ] 감사로그 + 알람 체계

## 8) 중요한 결론
- `AI Public Key + MPC Wallet ID`를 안다고 해서 자동으로 지갑 주인이 되면 안 된다.
- 소유권은 **등록 데이터가 아니라 서명 권한 정책**으로 결정되어야 한다.
- "선물"을 안전하게 만들려면, 기존 인간 권한을 반드시 끊고 AI 소유자 고정을 적용해야 한다.

## 동의하는 부분

문제 A~D 모두 실제 위험입니다. 그중 **문제 D가 AICW에서 가장 치명적**입니다. 우리가 이 대화에서 계속 다뤄온 문제와 정확히 같습니다. "AI에게 지갑을 선물했다"고 하면서 인간이 브리지 접근 권한을 계속 가지고 있으면, 그건 선물이 아니라 대여입니다. 해결책 7번 "권한 이전 절차, 기존 인간 계정 권한 제거"가 이걸 해결합니다.

즉시 적용(Must) 4가지도 전부 맞습니다. walletId 1:1 고정, 중복 차단, 소유자 검증, rate limit. 이것들 없이 프로덕션에 나가면 시나리오 1~3이 실제로 발생할 수 있습니다.

## 제가 추가로 짚을 부분

해결책 7번 "권한 이전 절차"가 가장 중요한데, 구현 방법이 명시되어 있지 않습니다. 구체적으로 말하면 이런 질문이 남습니다. 인간의 브리지 권한을 "끊는다"는 게 Mpcium 수준에서 가능한가. Mpcium의 event_initiator_pubkey를 교체하거나 폐기하는 절차가 있는가. 아니면 Predict 서버 수준에서 인간의 API 접근만 차단하는 것인가. 서버 수준 차단이면, 서버 관리자가 다시 열 수 있으므로 완전한 차단이 아닙니다.

이건 우리가 이전에 논의한 "issuer_pubkey 우회 문제"와 본질이 같습니다. 온체인에서 인간 서명을 차단하는 건 AICW 스마트컨트랙트가 이미 하고 있지만, 오프체인 브리지 접근까지 차단하는 건 별도로 설계해야 합니다.

---

## 해결책 6번에 대한 의견

"일일 한도, 허용 대상 제한"이 포함되어 있는데, 이건 AICW 철학과 충돌할 수 있습니다. 우리가 논의한 대로, AI 완전통제 지갑에서 인간이 설정하는 한도는 모순입니다. 이 스코프를 누가 설정하느냐가 중요합니다. 인간이 설정하면 AI 완전통제가 아니고, AI가 스스로 설정하면 괜찮습니다. Composer 2에게 이 구분을 명확히 해달라고 요청하는 것이 좋겠습니다.
